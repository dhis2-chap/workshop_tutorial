Part 1: Terminal

Let's start with the terminal. This is the foundation for everything else.

What is the Terminal?

A text-based interface for your computer

Instead of clicking...

ls

...type commands

The terminal is a text interface. Instead of clicking through folders, you type commands. It might feel unfamiliar at first, but it's powerful once you get used to it.

Opening the Terminal

  • Mac: Applications → Utilities → Terminal
  • Windows: Open WSL (you should have this installed)
  • Linux: Terminal app

Open your terminal now!

Let's all open our terminals. Mac users: Spotlight search for Terminal. Windows users: You should have WSL installed - search for WSL or Ubuntu in the start menu. Give people about 30 seconds to open their terminals.

Essential Commands

pwdWhere am I?
lsWhat's here?
cd folderGo into folder
cd ..Go up one level
mkdir nameCreate folder
touch fileCreate empty file
These are the six commands you'll use constantly. pwd - print working directory - tells you where you are ls - list - shows what's in the current folder cd - change directory - moves you around mkdir - make directory - creates a folder touch - creates an empty file

ls
DEMO TIME. First, run pwd. This shows your current location. You should see something like /home/username or /Users/username. Now run ls. This lists everything in your current folder. You might see Desktop, Documents, Downloads, etc.

Navigation Tips

  • Tab - autocomplete file/folder names
  • Up arrow - previous command
  • Ctrl+C - cancel/stop
Three tips that will save you time: Tab autocompletes names - try typing "cd work" then pressing Tab. Up arrow recalls your previous command. Ctrl+C stops whatever is running - useful if something hangs.

Part 2: Git & GitHub

Now let's talk about Git and GitHub. This is how you'll share and collaborate on code.

GitHub vs Git

GitHub

Website for sharing code

(like Google Drive for code)

Git

Tool to upload/download code

(runs in your terminal)

Common point of confusion: GitHub is the website, Git is the command-line tool. GitHub is where code lives online. Git is how you interact with it from your computer.

Key Concepts

  • Repository (repo) - A project folder on GitHub
  • Fork - Your own copy of a repo on GitHub
  • Clone - Download a repo to your computer
Three key terms: Repository - a project folder, contains all the files for a project. Fork - your personal copy on GitHub. Changes you make don't affect the original. Clone - downloading a repo to your computer so you can work on it.

The Workflow

  1. Fork on GitHub (get your own copy)
  2. Clone to your computer
  3. Edit files locally
  4. Commit (save a snapshot)
  5. Push (upload to GitHub)
This is the typical workflow you'll use: Fork creates your copy on GitHub. Clone downloads it. Edit files with any editor. Commit saves a snapshot with a message. Push uploads your changes back to GitHub.

Why This Matters

  • Backup - Your code is safe on GitHub
  • History - See what changed and when
  • Undo - Revert to previous versions
  • Share - Others can see and use your code
Why bother with all this? Your code is backed up - if your laptop dies, it's still on GitHub. You have full history - see exactly what changed and when. You can undo mistakes by reverting to earlier versions. And you can share your work with collaborators.

Step 1: Fork on GitHub

Go to:

github.com/dhis2-chap/chap-workshop-python

  1. Click "Fork" (top right)
  2. Select your account

Now you have: github.com/YOUR-USERNAME/chap-workshop-python

DEMO: Open the CHAP Python SDK repository. Show the Fork button in the top right. Click it, select your account. Now you have your own copy. Notice the URL changes to YOUR-USERNAME/chap-workshop-python. Give people time to do this - about 1-2 minutes.

Step 2: Clone to Your Computer


# Navigate to where you want the folder
cd ~

# Clone YOUR fork (replace YOUR-USERNAME)
git clone https://github.com/YOUR-USERNAME/chap-workshop-python.git

# Enter the folder
cd chap-workshop-python
DEMO: Now clone your fork. cd ~ takes you to your home directory. Replace YOUR-USERNAME with your actual GitHub username. This creates a chap-workshop-python folder with all the files. cd into it.

Step 3: Check Status


git status

Should say: "nothing to commit, working tree clean"

DEMO: git status is your best friend. It tells you what's changed, what's staged, and what branch you're on. Right now it should say "nothing to commit" because we haven't changed anything yet.

Step 4: Edit a File

Open README.md in any editor

Add a line, e.g.:


## My Notes
Testing git workflow!

Save the file

DEMO: Open README.md in VS Code or any text editor. Add a line at the bottom - anything you want. Save the file. If using VS Code, you can run "code ." to open the folder.

Step 5: Check Status Again


git status

Now shows: README.md modified (in red)

DEMO: Run git status again. Now it shows README.md as modified, in red. Red means "changed but not staged".

Step 6: Stage Changes


git add README.md
                

git status

Now shows: README.md staged (in green)

DEMO: Stage the file with git add. This marks it for inclusion in the next commit. Run git status again - now it's green, meaning "staged and ready to commit".

Step 7: Commit


git commit -m "Add my notes to README"

The message describes what you changed

DEMO: Commit with a message. The -m flag lets you write the message inline. Messages should describe what you changed - this helps you later. This creates a snapshot, but it's still only on your computer.

Step 8: Push to GitHub


git push

Check GitHub - your change is there!

DEMO: Push uploads your commit to GitHub. Now go to your fork on GitHub and refresh. You should see your change in the README!

Quick Reference


# Clone
git clone https://github.com/YOUR-USERNAME/chap-workshop-python.git
cd chap-workshop-python

# Edit README.md, then:
git status              # See what changed
git add README.md       # Stage changes
git commit -m "message" # Save snapshot
git push                # Upload to GitHub
Here's the quick reference. status, add, commit, push - that's the core workflow.

Common Issues

  • "Permission denied" - Run gh auth login
  • Wrong username in URL - Use YOUR fork URL
  • "Nothing to commit" - Did you save the file?
Common problems: Permission denied - you need to authenticate with gh auth login. Wrong URL - make sure you're cloning YOUR fork, not the original. Nothing to commit - make sure you saved your changes in the editor.
